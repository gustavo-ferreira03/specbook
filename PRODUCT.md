# Product

## Register

product

## Users

Independent developers and small software teams who need to document and verify web application behavior against a live application. They describe flows, watch an agent explore them, and keep the resulting behavior as durable documentation.

## Product Purpose

Specbook turns conversations about web application behavior into permanent, human-readable Specs with verification status, run history, and evidence. A user should be able to connect an application, describe a flow without writing test code, and return later to a clear record of what the application should do and whether it still does it.

## Brand Personality

Precise, calm, and trustworthy. The interface should feel like a well-kept technical notebook, focused on readable behavior and dependable status rather than the automation behind it.

## Anti-references

Specbook should not look like a code-first test runner, expose Robot Framework source, or resemble a crowded enterprise QA dashboard. Avoid decorative metrics, excessive cards, generic SaaS decoration, and any interface that hides failures or relies on color alone for status.

## Design Principles

- Keep the Spec, not its automation, at the center of the product.
- Make state and failures visible wherever work happens.
- Use familiar controls that stay out of the documentation workflow's way.
- Keep the path from conversation to evidence short and legible.
- Preserve core actions and context on narrow screens.

## Accessibility & Inclusion

Use readable contrast, visible keyboard focus, descriptive labels, text alongside status color, and reduced-motion behavior. Project, conversation, Spec, run, and evidence workflows must remain usable on mobile and desktop.
